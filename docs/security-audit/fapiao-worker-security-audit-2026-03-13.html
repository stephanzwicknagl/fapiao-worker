Executive Summary

The fapiao-worker application demonstrates a strong overall security posture with no critical or high-severity vulnerabilities identified. The codebase applies solid defensive practices including multi-layer file upload validation, per-request isolated temporary directories with guaranteed cleanup, cryptographically secure session identifiers, Excel macro execution disabled, and a well-hardened systemd service configuration. Two medium-severity gaps were identified: sensitive fapiao data (invoice numbers, amounts, company names) can accumulate indefinitely on disk when users abandon the categorization flow, and standard HTTP security headers (X-Content-Type-Options, X-Frame-Options, Content-Security-Policy) are not set. Three low-severity findings cover the Docker container running as root, no application-level rate limiting on the expensive /process endpoint, and the absence of a CI/CD pipeline for automated security scanning. Remediating the medium findings is recommended before expanding the user base, while the low findings represent incremental hardening measures.

Detailed Findings

M1 – Pending Sessions Have No Expiry Medium

Description: When the /process endpoint finds unmapped sellers, it persists the extracted fapiao data (invoice numbers, amounts, company names) plus the uploaded Excel template to /tmp/fapiao_pending/<uuid>/ and redirects the user to the categorization page. The cleanup in the /categorize finally block (line 308) only runs when the user completes the flow. If the user navigates away, closes the tab, or the browser crashes, the pending directory is never deleted. Over time this leaks financial document data on the server filesystem. In a shared or containerised environment this data may be readable by other processes.

Affected Code

File: app.py, Lines 34, 207–212

PENDING_DIR = Path(tempfile.gettempdir()) / 'fapiao_pending'

# In /process – state is saved but only cleaned up on completion:
uuid = secrets.token_urlsafe(16)
pending = PENDING_DIR / uuid
pending.mkdir(parents=True, exist_ok=True)
(pending / 'fapiaos.json').write_text(json.dumps(fapiaos), encoding='utf-8')
(pending / 'template.xlsx').write_bytes(template_path.read_bytes())

Recommended Fix

SESSION_TTL_SECONDS = 3600  # 1 hour

def _cleanup_stale_pending() -> None:
    """Remove pending session directories older than SESSION_TTL_SECONDS."""
    if not PENDING_DIR.exists():
        return
    now = time.time()
    for entry in PENDING_DIR.iterdir():
        if entry.is_dir() and (now - entry.stat().st_mtime) > SESSION_TTL_SECONDS:
            shutil.rmtree(entry, ignore_errors=True)
            app.logger.info('Cleaned up stale session: %s', entry.name)

# Call at start of /process (or via before_request with throttle)
_cleanup_stale_pending()

M2 – Missing HTTP Security Headers Medium

Description: The Flask application sets no HTTP security headers. Without X-Content-Type-Options: nosniff, browsers may interpret responses with incorrect MIME types, enabling MIME-sniffing attacks. Without X-Frame-Options: DENY, the application can be embedded in iframes and is vulnerable to clickjacking. Without a Content-Security-Policy, inline script injection (e.g. via a future XSS bug) has no browser-level backstop. Without Referrer-Policy, URLs visited on the app may leak in HTTP Referer headers to third parties.

Affected Code

File: app.py, Line 20 (Flask initialization – no after_request hook present)

app = Flask(__name__)
app.secret_key = os.environ['SECRET_KEY']
# No security headers are configured

Recommended Fix

@app.after_request
def set_security_headers(response):
    response.headers['X-Content-Type-Options'] = 'nosniff'
    response.headers['X-Frame-Options'] = 'DENY'
    response.headers['Referrer-Policy'] = 'no-referrer'
    response.headers['Content-Security-Policy'] = (
        "default-src 'self'; "
        "style-src 'self' 'unsafe-inline'; "
        "script-src 'self'; "
        "img-src 'self' data:; "
        "form-action 'self';"
    )
    return response

L1 – Docker Container Runs as Root Low

Description: The Dockerfile contains no USER directive. The gunicorn process and any file operations inside the container execute as root (UID 0). If an attacker exploits a vulnerability in the PDF parsing library (PyMuPDF) or the web framework to achieve code execution, they would have root-level access within the container. The systemd deployment path correctly runs as the unprivileged fapiao user; parity with Docker is missing.

L2 – No Application-Level Rate Limiting on /process Low

Positive Findings

  • Secret key from environment: app.secret_key = os.environ['SECRET_KEY'] — the application fails to start if the key is absent, preventing accidental deployment with a default or empty secret.
  • Multi-layer file upload validation: File uploads are validated by extension, browser-supplied MIME type, and werkzeug.utils.secure_filename — three independent layers before any file is written to disk.
  • Upload limits enforced: MAX_CONTENT_LENGTH = 50 MB and MAX_PDF_FILES = 100 are enforced at the Flask level, preventing memory exhaustion from oversized requests.
  • Isolated temp directories with guaranteed cleanup: Every request creates its own tempfile.mkdtemp() directory; a finally block unconditionally calls shutil.rmtree(), preventing cross-request data leakage even when exceptions occur.
  • Cryptographically secure session IDs with whitelist validation: Session UUIDs use secrets.token_urlsafe(16) and are validated against a strict character whitelist before any filesystem access — preventing path traversal via the UUID parameter.
  • Excel macro execution disabled: openpyxl.load_workbook(template, keep_vba=False) is used in all code paths, ensuring that malicious macros in uploaded .xlsx files cannot execute.
  • Generic error messages to users: Exceptions show only "An unexpected error occurred" to users while full stack traces are emitted to server logs via app.logger.exception().
  • Jinja2 auto-escaping enabled: Flask's default Jinja2 configuration auto-escapes all template variables, preventing reflected XSS from user-controlled data in error messages and form fields.
  • Systemd service hardening: The fapiao-worker.service file sets PrivateTmp=true, NoNewPrivileges=true, and runs under a dedicated unprivileged fapiao user — limiting blast radius in the event of exploitation.
  • Docker bound to localhost only: compose.yaml binds the port as 127.0.0.1:8000:8000, preventing direct external access and requiring a reverse proxy (nginx) for TLS termination and additional hardening.
